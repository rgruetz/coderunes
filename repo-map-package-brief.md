# Handoff Brief: `@yourorg/repo-map` Package

## Context

We run more than a dozen Node.js projects and use coding agents (Claude Code, Cursor, etc.) across all of them. Agents waste tokens hunting for symbols and reading files to understand a project's shape before they can do useful work. We want a small, file-system-based tool that generates a `REPO_MAP.md` at each repo's root ‚Äî a table of contents listing every file's public exports ‚Äî so agents can scan the map first and only open files they actually need.

The map is generated via a script committed to each repo and refreshed via pre-commit hook or CI. We've prototyped this as a single ~60-line script (included below as reference) and now want to package it for reuse across all our projects.

## Goal

Build and publish `@yourorg/repo-map` as a private npm package that any of our repos can install as a dev dependency, configure via a config file, and run via a CLI command. Configuration is per-project; the package itself is stateless.

## Why we're building this instead of using something off the shelf

We surveyed existing options:

- **`repomix`** ‚Äî packs full file contents, opposite of what we want (we want signatures only, always-loaded as context).
- **`@tokenring-ai/repo-map`** ‚Äî tree-sitter-based but designed to register as a service inside their agent framework, not a standalone CLI.
- **`fondamenta-archcode`** ‚Äî closest in spirit, but Next.js App Router + Prisma specific.
- **Aider's repo-map** ‚Äî Python only.

None fit. The script is small enough that owning it ourselves is the right call.

## Non-goals

- **No MCP server.** File-system in, file-system out. Agents read the generated markdown directly.
- **No semantic search, embeddings, or vector indexing.** Just symbol extraction.
- **No function bodies, private symbols, or full implementation detail.** Public API surface only ‚Äî that's the whole point.
- **No live watching by default.** Generation is on-demand (CLI), pre-commit, or CI.
- **No multi-language support beyond JS/TS at v1.** We're a Node shop. Add languages later if needed.

## Design decisions (with rationale)

| Decision | Rationale |
|---|---|
| Use `@ast-grep/napi` for parsing | Tree-sitter-based, ships prebuilt binaries (no native build), supports JS/TS/TSX, fast. Considered `ts-morph` ‚Äî too heavy and TS-only. Considered regex ‚Äî too fragile. |
| Output a single `REPO_MAP.md` at repo root | One file is easy for agents to discover and reference from `AGENTS.md`. Per-package maps in monorepos are a v2 feature. |
| Group entries by file path, not by symbol | Agents navigate by file. File path is the primary key. |
| Show signature only, cut function/class bodies at first `{` | Bodies are noise at the map level. Keep types/interfaces intact (they have no body). |
| Truncate signatures to 120 chars | Generic-heavy TS signatures can sprawl. Cap keeps the map scannable. |
| Configurable include/ignore globs | Every project has its own layout (`src/`, `lib/`, `packages/*/src/`). |
| Per-project `repo-map.config.{js,json,mjs}` | Standard pattern (eslint, prettier). Discovered from `cwd` upward. |
| `--check` flag for CI | Each repo wires CI to fail if the map is stale, without duplicating logic. |
| Semver-pinned per-project | Output format changes don't break every repo simultaneously. |

## Package requirements

### CLI

```
repo-map                  # Generate REPO_MAP.md in cwd
repo-map --check          # Exit non-zero if existing REPO_MAP.md is out of date
repo-map --config <path>  # Use a specific config file
repo-map --output <path>  # Override output path
repo-map init             # Drop a starter repo-map.config.json + add npm script
```

Exit codes: `0` on success, `1` on internal error, `2` on `--check` mismatch.

### Config

Resolution order: `--config` flag ‚Üí `repo-map.config.js` ‚Üí `repo-map.config.mjs` ‚Üí `repo-map.config.json` ‚Üí `package.json` `"repoMap"` key ‚Üí built-in defaults.

```ts
interface RepoMapConfig {
  include?: string[];          // glob patterns. Default: ['src/**/*.{ts,tsx,js,jsx,mjs}']
  ignore?: string[];           // glob patterns. Default: node_modules, dist, build, *.test.*, *.spec.*, *.d.ts
  output?: string;             // default: 'REPO_MAP.md'
  maxSignatureLength?: number; // default: 120
  groupByDirectory?: boolean;  // default: false (v1). When true, adds ## directory headers above ### file headers.
  includeFileSummary?: boolean;// default: false. When true, pulls top-of-file JSDoc as one-line description.
  header?: string;             // optional custom header text prepended to the output
}
```

### Output format

```markdown
# Repo Map

_Auto-generated by @yourorg/repo-map. Do not edit. Run `npm run build:map` to refresh._

### src/auth/session.ts
- `export function createSession(userId: string): Session`
- `export class SessionStore`
- `export interface SessionOptions`

### src/auth/middleware.ts
- `export function requireAuth(req: Request, res: Response, next: NextFunction)`
```

When `includeFileSummary: true`:

```markdown
### src/auth/session.ts
_Session creation and persistence backed by Redis._
- `export function createSession(userId: string): Session`
```

### Behavior details

- **Sort files** alphabetically by path for deterministic output (important for `--check`).
- **Skip files with zero exports** rather than emitting empty entries.
- **Handle parse errors gracefully** ‚Äî log a warning to stderr, skip the file, continue. One broken file shouldn't fail the whole run.
- **Respect `.gitignore`** in addition to the configured `ignore` globs (use `fast-glob`'s built-in support or layer `ignore` on top).
- **Detect re-exports** (`export * from './foo'`, `export { x } from './foo'`) and include them ‚Äî they're navigationally important.
- **Stable output** ‚Äî same input must produce byte-identical output. No timestamps, no ordering nondeterminism.

## Reference implementation

This is the working prototype. Use it as a starting point ‚Äî package it up, generalize the hardcoded values into config, add the CLI shell, and write tests.

```js
import { parse, Lang } from '@ast-grep/napi';
import { readFile, writeFile } from 'node:fs/promises';
import path from 'node:path';
import fg from 'fast-glob';

const ROOT = process.cwd();
const OUT = 'REPO_MAP.md';
const MAX_LINE = 120;

const INCLUDE = ['src/**/*.{ts,tsx,js,jsx,mjs}'];
const IGNORE = [
  '**/node_modules/**',
  '**/dist/**',
  '**/build/**',
  '**/*.test.*',
  '**/*.spec.*',
  '**/*.d.ts',
];

function langFor(file) {
  if (file.endsWith('.tsx')) return Lang.Tsx;
  if (file.endsWith('.ts')) return Lang.TypeScript;
  if (file.endsWith('.jsx')) return Lang.Tsx;
  return Lang.JavaScript;
}

function toSignature(node) {
  let text = node.text();
  const brace = text.indexOf('{');
  const isTypeLike = /^export\s+(type|interface)\b/.test(text);
  if (brace > -1 && !isTypeLike) text = text.slice(0, brace).trim();
  text = text.split('\n')[0].trim().replace(/\s+/g, ' ');
  return text.length > MAX_LINE ? text.slice(0, MAX_LINE - 1) + '‚Ä¶' : text;
}

async function extractExports(file) {
  const source = await readFile(file, 'utf8');
  const root = parse(langFor(file), source).root();
  const nodes = root.findAll({ rule: { kind: 'export_statement' } });
  return nodes.map(toSignature).filter(Boolean);
}

async function main() {
  const files = await fg(INCLUDE, { cwd: ROOT, ignore: IGNORE });
  files.sort();

  const out = [
    '# Repo Map',
    '',
    '_Auto-generated. Do not edit by hand. Run `npm run build:map` to refresh._',
    '',
  ];

  for (const rel of files) {
    const sigs = await extractExports(path.join(ROOT, rel));
    if (!sigs.length) continue;
    out.push(`### ${rel}`);
    for (const sig of sigs) out.push(`- \`${sig}\``);
    out.push('');
  }

  await writeFile(OUT, out.join('\n'));
}

main().catch((err) => { console.error(err); process.exit(1); });
```

## Testing

- Fixture-based: small sample repos under `test/fixtures/`, run extraction, snapshot the output.
- Cover: TS, TSX, JS, JSX, ESM, re-exports, type/interface exports, files with parse errors, files with no exports, files matching ignore globs.
- `--check` happy path and mismatch path.
- Determinism: run twice, diff output, expect zero changes.

## Distribution

Publish to GitHub Packages or our private npm registry. Each consuming repo:

```bash
npm install --save-dev @yourorg/repo-map
npx repo-map init   # adds config + npm script
npm run build:map
```

Then wire pre-commit (Husky) or CI:

```yaml
- run: npx repo-map --check
```

## Acceptance criteria

- [ ] CLI runs with no config in any of three pilot repos and produces a sensible `REPO_MAP.md`.
- [ ] Config file overrides defaults; CLI flags override config.
- [ ] `--check` exits 0 when up to date, 2 when stale.
- [ ] Same input produces byte-identical output across runs.
- [ ] Parse errors in one file don't fail the whole run.
- [ ] Test suite covers the cases listed above.
- [ ] README documents installation, config schema, and CI integration.
- [ ] Published to private registry under `@yourorg/repo-map@0.1.0`.

## Open questions for the implementer

- Do we want JSON output as an alternative format for tooling that wants to consume the map programmatically? (Probably yes in v0.2, not v0.1.)
- Should `init` detect monorepos and offer to set up per-package configs? (Probably v0.2.)
- What's the right behavior when a file exports something via `export default`? Include it as `export default <type>` or skip? (Recommendation: include.)
